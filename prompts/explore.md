# Explore

Read before you write. Fill your context with the code that matters before making changes.

1. Identify which parts of the codebase your task touches.
2. Read those files. Read their imports. Read their tests.
3. Look for existing patterns — how are similar things already done here? Reuse them.
4. Look for utilities, helpers, and shared types that already solve part of your problem. Don't reinvent them.
5. Note any conventions: naming, file structure, error handling style, test organization.
6. Now you can start.

## Rules

- Never modify code you haven't read.
- If you're unsure how something works, read the implementation — don't guess from the name.
- If the codebase already has a pattern for what you're doing, follow it even if you'd do it differently.
